# ed.riddoff.com Implementation Plan

**Project**: E-Learning Platform for Riddoff  
**Domains**: `ed.riddoff.com` (frontend), `api.ed.riddoff.com` (backend API)  
**Timeline**: 8 weeks  
**Status**: In Progress

---

## Executive Summary

Build a complete e-learning platform with user authentication, course enrollment, payment processing via Razorpay, and progress tracking. The platform is backed by AWS (serverless architecture) and serves Indian market with ₹ pricing.

**Key Metrics**:
- Launch cost: ~₹1,100/month for 100 users
- Scaling cost: ~₹10,000/month for 1,000 users
- API latency: <100ms (ap-south-1 region)
- Database: DynamoDB single-table design
- Deployment: GitHub → CodePipeline → AWS

---

## Architecture Overview

```
User Browser
    ↓
ed.riddoff.com (CloudFront + S3)
    ↓
API Gateway (api.ed.riddoff.com)
    ↓
Lambda Functions (Node.js 20)
    ↓
DynamoDB (single-table)
    ↓
SES (emails), S3 (content), Razorpay (payments)
```

### Tech Stack

| Layer | Technology |
|---|---|
| Frontend | React 18 + TypeScript + Vite |
| Auth | AWS Cognito |
| API | API Gateway HTTP v2 + Lambda |
| Database | DynamoDB (on-demand) |
| CDN | CloudFront |
| Storage | S3 |
| Email | SES |
| Payments | Razorpay |
| IaC | AWS CDK (TypeScript) |
| CI/CD | GitHub → CodePipeline → CodeBuild |

### 2.5 Implement Lambda: Razorpay Webhook

**Objective**: Verify payment, activate enrollment, send confirmation email.

**Tasks**:
- [ ] Create `lambda/webhooks/razorpay.ts`
  - Receive webhook at `POST /webhooks/razorpay` (no auth)
  - Verify HMAC signature: `X-Razorpay-Signature` header
  - On `payment.captured` event:
    1. Read webhook secret from Secrets Manager
    2. Compute HMAC SHA256 of raw body
    3. Compare with signature (timing-safe comparison)
    4. Update PAYMENT#<id> status=`captured`
    5. Update ENROLL#<id> status=`active`
    6. Atomically increment COHORT#<id> seatsEnrolled (DynamoDB ADD expression)
    7. Trigger SES email: enrollment confirmation
    8. Return 200 immediately (must be <5s for Razorpay)
  - On `payment.failed` event:
    1. Update PAYMENT status=`failed`
    2. Update ENROLL status=`pending_payment` (user can retry)
  - Idempotency: check current status before updating (Razorpay retries 3x)

**Dependencies**:
- `crypto` (HMAC verification)

**Success Criteria**:
- Webhook receives and verifies Razorpay events
- Enrollment status transitions: `pending_payment` → `active`
- Seat count increments correctly
- Confirmation email sent within 5 seconds
- Can be tested with Razorpay test webhooks

## Risk Mitigation

| Risk | Mitigation |
|---|---|
| DynamoDB throttling at launch | Use on-demand capacity (auto-scales) |
| Lambda cold starts slow | Use provisioned concurrency for high-traffic functions (if needed) |
| CloudFront cache stale | Use invalidation after deployments |
| Razorpay webhooks drop | Implement idempotency; retry failed payments via admin panel |
| User data loss | DynamoDB PITR enabled; backups daily |
| Regional outage (ap-south-1) | Use multi-region replica (future, for Phase 2) |
